Module 4 — Regression, Evidence & Testsystemen

In deze module ga je testen zien als een systeem, niet als losse acties. Niet: “ik test iets”, maar: “ik bouw vertrouwen op in verandering”.

1. Wat is regressietesten écht?

Regressietesten = vertrouwen behouden tijdens verandering

Elke wijziging kan iets breken. Regressietesten zorgen ervoor dat wat eerst werkte, blijft werken.

2. Smoke test vs regression set

  • Smoke test: werkt het systeem globaal nog?
  • Regression set: werken de kritieke functies nog?

Je gaat nu beide ontwerpen.

3. Oefening A — Smoke test set

Kies één oefensite (bijv. Sauce Demo, ParaBank of The Internet).

  1. Definieer 5 checks die altijd moeten werken.
  2. Dit zijn geen details, maar systeemchecks.

Voorbeeld (webshop)

  • Login werkt
  • Productpagina laadt
  • Product in winkelwagen
  • Checkout bereikbaar
  • Bevestigingspagina zichtbaar

4. Oefening B — Mini regression set

Ontwerp een regressieset van 8–10 tests:

  • 3 kritieke flows
  • 3 negatieve scenario’s
  • 2 datavarianten
  • 1 randgeval
Dit is jouw “beschermlaag” tegen regressie.

5. Evidence verzamelen

Testen zonder bewijs is mening. Testen met bewijs is informatie.

  • Screenshot (UI)
  • Console log
  • Network tab
  • Request/response
  • Timestamps

6. Oefening C — Evidence based testing

  1. Kies 1 bug of issue
  2. Verzamel 3 soorten bewijs
  3. Maak er één professioneel testdossier van

Structuur

  • Context
  • Reproduceerstappen
  • Bewijs
  • Impact
  • Risico bij niet oplossen

7. Denken in systemen

Stel jezelf bij elke test de vragen:

  • Wat kan hier kettingreacties veroorzaken?
  • Wat is afhankelijk van wat?
  • Wat breekt als dit verandert?
Testen = systeemdenken, niet foutzoeken.

8. Deliverable

  • 1 smoke test set (5 checks)
  • 1 regression set (8–10 tests)
  • 1 evidence dossier
  • 1 reflectie: “Waar zit systeemrisico?”

9. Reflectie

  1. Wat gaf jou het meeste vertrouwen in het systeem?
  2. Wat voelde als schijnzekerheid?
  3. Waar zou automatisering zinvol zijn — en waar niet?
