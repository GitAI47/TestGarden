Module 2 — Testlevels & Smoke Testing

In deze module ontdek je hoe software testen in verschillende lagen gebeurt, en wat de rol is van de bekende smoke test. We gaan van overzicht naar praktijk, zodat jij begrijpt hoe jouw testwerk onderdeel is van het grotere geheel.

1. Wat zijn testlevels?

Software wordt niet in één keer getest. Elk onderdeel wordt op een ander niveau gecontroleerd. Deze niveaus noemen we testlevels.

  • Unit Testing — kleine stukjes code, getest door ontwikkelaars.
  • Integratietests — werkt alles samen zoals bedoeld?
  • Systeemtests — werkt de hele applicatie als geheel?
  • Acceptatietests — werkt het voor de gebruiker en de opdrachtgever?

Voorbeeld

In een webshop:
• unit test → werkt de berekening van korting? • integratietest → werkt winkelwagen + voorraad samen? • systeemtest → werkt het bestelproces end-to-end? • acceptatietest → vindt de klant de flow logisch en bruikbaar?

2. Testtypes: functioneel & niet-functioneel

Naast testlevels heb je ook testtypes. Dit beschrijft wát je test.

  • Functioneel testen — doet de functie wat hij moet doen?
  • Niet-functioneel testen — snelheid, veiligheid, gebruiksgemak, stabiliteit…

In Testgarden richten we ons vooral op functioneel testen, zoals smoke tests en exploratory testing — de basis voor elke startende tester.

3. Wat is een Smoke Test?

Een smoke test is een korte, snelle check om te zien of het systeem “ongeveer gezond” is na een nieuwe release, update of deploy.

Het is de digitale variant van: “gaat de rookmelder af?” Als iets fundamenteels stuk is, wil je dat meteen weten.

Waarom smoke tests?

  • Ze zijn snel en geven direct duidelijkheid.
  • Ze voorkomen dat testers tijd verspillen aan een kapotte build.
  • Ze geven een duidelijke GO/NO-GO voor verdere testen.

Voorbeeld van een smoke test

  • Laadt de applicatie?
  • Kan de gebruiker inloggen?
  • Werkt de belangrijkste flow (bijv. item toevoegen)?
  • Zijn er geen 404- of 500-fouten?

4. Voorbeeld Smoke Test Checklist

Hieronder een eenvoudige checklist die je kunt gebruiken voor elke webapplicatie:

  • [ ] Is de website bereikbaar? (Status 200)
  • [ ] Kan een testgebruiker inloggen?
  • [ ] Werkt de belangrijkste functionaliteit?
  • [ ] Werken de belangrijkste links en knoppen?
  • [ ] Zijn er geen grote fouten zichtbaar (404, 500, crash)?
  • [ ] Werkt het ook in een andere browser of op mobiel?

Je sluit een smoke test af met een eenvoudige GO / NO-GO.

5. Praktijkopdracht: Maak en voer je eigen smoke test uit

Je gaat nu zelf een smoke test ontwerpen én uitvoeren. Dit is precies wat testers in echte teams dagelijks doen.

Opdracht

  1. Kies een demo-webapp (door de docent, of een openbare demo).
  2. Maak je eigen smoke test checklist (minimaal 6 checks).
  3. Voer de checklist uit.
  4. Noteer per stap: Pass of Fail.
  5. Schrijf een korte conclusie: is het GO of NO-GO?
Tip: Zorg dat de stappen eenvoudig blijven. Een smoke test is geen volledige test — het is een snelle gezondheidsscan.

6. Reflectie

Denk na over je test:

  • Welke check was het belangrijkst?
  • Welk probleem zou het grootste risico vormen?
  • Hoe zou je checklist verbeteren voor de volgende keer?
